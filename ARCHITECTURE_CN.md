## 1. 项目概述

### 1.1 什么是 OpenModelPool？

OpenModelPool 是一个**临时 Token 银行 + 极客共享网络**。默认情况下，它是一个**纯本地的个人模型代理**（个人版模式）。只有当用户配置了 Provider Token、开启额度管理、并且系统检测到本月有闲置额度时，才会提示用户**主动选择加入共享网络**（共享版模式）。Provider Token **永不上传**到任何服务器。

### 1.2 核心理念

```
┌─────────────────────────────────────────────────────────────┐
│                   双重叙事                                      │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  临时 Token 银行叙事：                                        │
│    你的 GPT-4o 额度本月只用了 60%？剩余 40% 过期浪费            │
│    → 存入 OpenModelPool 网络                                  │
│    → 别人使用后你获得贡献积分                                   │
│    → 未来你需要 Claude-3 时用积分取回等价资源                    │
│                                                              │
│  BT 类比叙事（仅共享版模式）：                                   │
│    BitTorrent:    对等节点共享文件    → 下载/上传               │
│    OpenModelPool: 对等节点共享服务    → 消费/贡献               │
│                                                              │
│  ⚠️ 关键区分：                                                │
│    个人版（默认）：纯本地代理，不联网                            │
│    共享版（按需加入）：P2P 共享，生成 Node ID                    │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 关键原则

| 原则 | 描述 |
|--------|------|
| **个人版优先** | 默认离线个人代理；网络是可选加入的 |
| **去中心化** | 无中心服务器；节点通过 DHT 发现和路由 |
| **开放贡献** | 任何节点都可以将 API 访问权限贡献到共享池 |
| **贡献积分** | 不可提现、不可交易，绑定 Node ID |
| **基于信任的路由** | 声誉系统确保服务质量 |
| **故障关闭安全** | 授权链在任何错误时安全地失败关闭 |
| **仅 4 种密钥** | Proxy API Key / Guest Proxy Key / Public Global Key / Provider Key（`mk_*` 已废弃） |

## 2. 网络拓扑架构

### 2.1 Kademlia DHT — 256 位哈希空间

OpenModelPool 实现了一个 Kademlia 风格的分布式哈希表，参数如下：

```
┌─────────────────────────────────────────────────────────────┐
│                  DHT 配置                                          │
├─────────────────────────────────────────────────────────────┤
│  哈希空间:     256 位 (SHA-256)                                   │
│  距离度量:     XOR 度量                                            │
│  K-Bucket 大小: k = 20（高冗余对抗节点流动）                       │
│  桶数量:       256（每比特一个）                                   │
│  查找 α:       10（P50 延迟约 0.3 秒）                            │
│  查找 β:       3（终止条件）                                       │
│  刷新间隔:     10 分钟                                             │
│  查询超时:     10 秒                                               │
│  记录 TTL:     48 小时                                             │
│  Protocol ID:    /openmodelpool/kad/1.0.0                   │
└─────────────────────────────────────────────────────────────┘
```

**为什么选择 256 位而非 BitTorrent 的 160 位？**
- BitTorrent uses SHA-1 (160-bit) — sufficient for ~8M nodes
- 256 位提供 2^96 倍的更大地址空间
- Matches IPFS/Kademlia standard, enabling future interoperability

### 2.2 节点哈希与距离

```go
// NodeHash — DHT 环中的 256 位位置（源码：dht.go）
type NodeHash [HashSizeBytes]byte  // HashSizeBytes = 32

func ComputeNodeHash(nodeID string) NodeHash {
    return sha256.Sum256([]byte(nodeID))
}

func XORDistance(a, b NodeHash) *big.Int {
    aInt := a.BigInt()
    bInt := b.BigInt()
    return new(big.Int).Xor(aInt, bInt)
}
```

### 2.3 K-Bucket 结构

```
K-Bucket Organization (k=20):

Bucket 0:   nodes with common prefix length 0   (distance: 2^255 ~ 2^256)
Bucket 1:   nodes with common prefix length 1   (distance: 2^254 ~ 2^255)
...
Bucket i:   nodes with common prefix length i   (distance: 2^(255-i) ~ 2^(256-i))
...
Bucket 255: nodes with common prefix length 255 (distance: 2^0 ~ 2^1)

每个桶最多容纳 20 个节点，按最近可见时间排序。
```

### 2.4 GossipSub 消息传播

```
┌──────────────────────────────────────────────────────────┐
│              GossipSub 网状网络                                    │
│                                                           │
│    ┌─────┐     ┌─────┐     ┌─────┐                      │
│    │ N1  │─────│ N2  │─────│ N3  │                      │
│    └──┬──┘     └──┬──┘     └──┬──┘                      │
│       │           │           │                           │
│       ▼           ▼           ▼                           │
│    ┌─────┐     ┌─────┐     ┌─────┐                      │
│    │ N4  │─────│ N5  │─────│ N6  │                      │
│    └─────┘     └─────┘     └─────┘                      │
│                                                           │
│  网格度:       6（可接受范围：4-12）                               │
│  扇出:         10                                                │
│  Gossip 间隔:  3 秒                                                │
│  传播:         8 秒内 99%（1 万节点）                              │
│  带宽:         每节点约 1.2 Mbps                                  │
└──────────────────────────────────────────────────────────┘
```

**消息类型：**
- Model status updates (capability changes)
- Inference queue depth announcements
- Reputation score broadcasts
- Node join/leave notifications
- Contribution record synchronization

### 2.5 节点发现

```
节点发现流程：

1. 引导阶段
   ┌─────────────────────────────────────┐
   │ 新节点联系引导节点               │
   │ → 获取初始对等节点列表            │
   │ → 加入 DHT 环                     │
   └─────────────────────────────────────┘

2. DHT 遍历
   ┌─────────────────────────────────────┐
   │ 迭代查找（α=10 并行）             │
   │ → 查询最近的已知节点               │
   │ → 发现新的邻居                     │
   │ → 填充 k-buckets                  │
   └─────────────────────────────────────┘

3. 对等节点交换 (PEX)
   ┌─────────────────────────────────────┐
   │ 基于 Gossip 的对等节点交换          │
   │ → 通过 Gossip 共享已知对等节点      │
   │ → 维护网状连接                      │
   └─────────────────────────────────────┘
```

**AutoNAT 角色判定：**
```go
func determineNodeMode(host host.Host, autonat autonat.AutoNAT) dht.ModeOpt {
    reachability := autonat.Status()
    if reachability == network.ReachabilityPublic {
        return dht.ModeServer  // 公网: DHT 服务器 + 推理服务
    }
    return dht.ModeClient      // NAT 后: 仅 DHT 客户端
}
```

func ClassifyKey(key string) KeyType {
    switch {

    case strings.HasPrefix(key, "sk-guest-"):
        return KeyTypeGuest
    case strings.HasPrefix(key, "sk-"):
        return KeyTypeProxy

## 4. 能力声明与服务广播

### 4.1 概念

每个节点广播其能提供的 AI 模型服务——类似于 BitTorrent 的 `bitfield` 消息声明对等节点持有哪些数据块。

```
┌─────────────────────────────────────────────────────────────┐
│              BitTorrent ↔ OpenModelPool 映射                      │
├─────────────────────────────────────────────────────────────┤
│  BitTorrent:  "我持有数据块 [0,2,3,6,7]"                          │
│  OpenModelPool: "我能提供 [gpt-4o, claude-3, deepseek]"           │
│                                                              │
│  BitTorrent:  上传带宽限制                                         │
│  OpenModelPool: API 配额 / 速率限制                                │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 CapabilityClaim 数据结构

```go
// CapabilityClaim — 节点能提供的服务
type CapabilityClaim struct {
    NodeID      string           `json:"node_id"`
    Timestamp   time.Time        `json:"timestamp"`
    ExpiresAt   time.Time        `json:"expires_at"`
    
    // 模型能力
    Models      []ModelCapability `json:"models"`
    
    // 节点资源
    MaxConcurrent int            `json:"max_concurrent"`  // 最大并行请求数
    RateLimit     int            `json:"rate_limit"`       // 每分钟请求数
    QuotaUsed     int64          `json:"quota_used"`       // 已消耗配额
    QuotaLimit    int64          `json:"quota_limit"`      // 总配额
    
    // 签名
    Signature   []byte           `json:"signature"`        // Ed25519 签名
}

// ModelCapability — 每个模型的能力声明
type ModelCapability struct {
    Provider    string    `json:"provider"`     // "openai", "anthropic"
    Model       string    `json:"model"`        // "gpt-4o", "claude-3-opus"
    Version     string    `json:"version"`      // 模型版本/日期
    Available   bool      `json:"available"`    // 当前是否可用？
    AvgLatency  int64     `json:"avg_latency"`  // 平均延迟（毫秒）
    SuccessRate float64   `json:"success_rate"` // 历史成功率
}
```

## 7. 虚假能力防御

### 7.1 问题描述

节点可能声称可以提供某个模型的服务，但实际上无法提供——这可能是由于配置错误、API 密钥过期或恶意行为。

### 7.2 多层防御

```
虚假能力防御系统：

┌─────────────────────────────────────────────────────────────┐
│                                                               │
│  第一层：主动探测                                              │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ 定期发送 1-token 测试请求                             │     │
│  │ • 成本: 每次探测约 $0.00001                          │     │
│  │ • 检测: 过期密钥、速率限制、离线节点                 │     │
│  └─────────────────────────────────────────────────────┘     │
│                                                               │
│  Layer 2: 交叉验证                                    │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ 多个节点独立验证同一声明                              │     │
│  │ • 至少需要 3 个独立验证                               │     │
│  │ • 差异触发调查                                        │     │
│  └─────────────────────────────────────────────────────┘     │
│                                                               │
│  第三层：惩罚机制                                              │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ 渐进式执行:                                           │     │
│  │ • 成功率 < 70%：发出警告                             │     │

│  │ • 成功率 < 30%：隔离（从池中移除）                   │     │
│  │ • 成功率 < 10%：封禁（全局广播）                     │     │
│  └─────────────────────────────────────────────────────┘     │
│                                                               │
│  第四层：全局广播                                              │

│  └─────────────────────────────────────────────────────┘     │
│                                                               │
└─────────────────────────────────────────────────────────────┘

### 7.3 成本收益分析

```
为什么虚假声明是不理性的：

虚假声明的成本：
  • 1-token 探测成本：约 $0.00001
  • 检测时间：< 5 分钟
  • 惩罚：永久封禁 + 声誉损失
  • 恢复：需要 30 天以上的诚实服务

虚假声明的收益：
  • 临时的路由优先级（检测到后即被移除）
  • 无实际收入（无法服务真实请求）

结论：成本 >> 收益 → 自我执行的诚实机制
```

│ Leecher（下载者）     │ 消费但不贡献的节点                              │
│                       │（声誉系统会抑制此行为）                          │
├───────────────────────┼─────────────────────────────────────────────┤
│ Pieces（数据块）      │ 模型请求/响应单元                               │
├───────────────────────┼─────────────────────────────────────────────┤
│ Bitfield（位域）      │ CapabilityClaim（模型可用性）                   │
├───────────────────────┼─────────────────────────────────────────────┤
│ Have 消息             │ 能力更新广播（GossipSub）                       │
├───────────────────────┼─────────────────────────────────────────────┤
│ Choking 算法          │ 动态容量控制 + 优先队列                         │
│                       │（基于贡献比率）                                 │
├───────────────────────┼─────────────────────────────────────────────┤
│ Tit-for-Tat（以牙还牙）│ 声誉系统（贡献追踪）                            │
├───────────────────────┼─────────────────────────────────────────────┤
│ 上传/下载比率         │ TokenBudget / TokenUsed                         │
│                       │（按模型细粒度追踪）                              │
├───────────────────────┼─────────────────────────────────────────────┤
│ SHA1 数据块哈希       │ Ed25519 签名 + SHA-256 哈希                     │
├───────────────────────┼─────────────────────────────────────────────┤
│ Optimistic unchoke    │ 为新/未测试节点预留槽位                         │
├───────────────────────┼─────────────────────────────────────────────┤
│ Rarest-first          │ 路由到负载最低的可用节点                        │
├───────────────────────┼─────────────────────────────────────────────┤
│ Snubbing（忽略）      │ 慢节点检测 + 替换                               │
└───────────────────────┴─────────────────────────────────────────────┘
```

### 10.2 关键差异

| 方面 | BitTorrent | OpenModelPool |
|--------|------------|---------------|
| **内容** | 静态文件（不可变） | 动态服务（有状态） |
| **延迟** | 可容忍（分钟级） | 敏感（毫秒级） |
| **验证** | SHA1 哈希（确定性） | API 响应（非确定性） |
| **激励** | 软性（以牙还牙） | 硬性（声誉 + 未来代币） |
| **流动率** | 高（节点加入/离开） | 中等（节点更稳定） |
| **拓扑** | 网格（非结构化） | DHT（结构化 + Gossip） |

### 10.3 核心洞见

```
BitTorrent:
  "我有文件块 #42，我会上传给你"
  → 静态、可验证、一次性传输

OpenModelPool:
  "我能提供 GPT-4o 服务，我会中继你的请求"
  → 动态、概率性、持续性服务

核心挑战：服务质量是非确定性的。
解决方案：声誉系统 + 主动探测 + 交叉验证。
```
